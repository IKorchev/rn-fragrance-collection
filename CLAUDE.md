# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A React Native (Expo) mobile app for tracking a personal fragrance/perfume collection. Users sign in with Google or Apple, search a scraped fragrance database (or add a fragrance manually if the catalog doesn't have it), add items to their collection, track how many times they've worn each one, and use a weighted-random "picker" to choose what to wear. Auth and app data live in Supabase (Postgres + Supabase Auth, with Row Level Security guarding per-user rows) â€” the same Supabase project as the standalone `fragrance-db` scraper/catalog repo. Search queries the `fragrances` catalog table directly via the `search_fragrances` Postgres RPC (live debounced search plus a brand filter), and items added from search or the Most Worn leaderboard populate `user_fragrances.fragrance_id`, the FK to the canonical catalog row. **The catalog is deliberately name/brand/image only**: Parfumo serves fabricated data (note pyramids, and by extension ratings/votes/gender/etc. can't be trusted) to scraper-flagged traffic, so all other scraped catalog metadata was dropped (migrations `drop_poisoned_fragrance_notes` + `drop_scraped_metadata_add_wear_leaderboard`; only `image_url` â€” our own R2 CDN â€” and `source_url` â€” re-scrape dedup key â€” survive beyond name/brand). Don't re-add *catalog* notes/rating/gender features until `fragrance-db` has a clean re-scrape â€” this is separate from the personal per-item `rating`/`notes` columns on `user_fragrances` (user-entered on the detail sheet, exempt from the ban since they aren't scraped). The old scraped Top-100 lists (`top_fragrances`) are gone too; the Discover tab (the app's landing tab) instead shows a community "Most Worn" top-100 leaderboard computed from real wear data (`wear_events` + the `top_worn_fragrances` SECURITY DEFINER RPC, windowed day/week/month/year/all, `EXECUTE` revoked from `anon`). The `discover_fragrances` RPC (random catalog slice) is currently unused â€” the old home-screen Discover feed and its `useCatalog()` hook were removed when the Home tab was cut.

Release-readiness pieces live alongside the core app: `supabase/functions/delete-account` (in-app account deletion, invoked from the profile screen â€” required by both app stores), a daily wear-reminder push with a per-device opt-out (`user_push_tokens.reminders_enabled`, toggled in the profile screen), and `docs/privacy-policy.md` (host publicly and link from both store listings).

## Commands

There is no unit-test suite or linter configured in this project â€” do not invent `npm test`/`npm run lint` commands. There IS a type checker: run `yarn typecheck` (`tsc --noEmit`) after changes. For on-device verification there are Maestro E2E flows (load the `maestro-e2e` skill).

```bash
yarn start           # expo start
yarn android         # expo run:android (local dev-client build)
yarn ios             # expo run:ios (local dev-client build)
yarn web             # expo start --web
yarn typecheck       # tsc --noEmit
```

The project is TypeScript-only (`strict: true`, `tsconfig.json` extends `expo/tsconfig.base`; NativeWind `className` props are typed via `nativewind-env.d.ts`). `src/lib/database.types.ts` holds Supabase-generated DB types â€” regenerate via the Supabase MCP (`generate_typescript_types`) after schema changes; `src/lib/queries.ts` and `src/contexts/auth-context.tsx` derive their row types (`CatalogFragrance`, `TopFragrance`, `UserFragrance`) from it via the `Tables<"...">` helper. Note: `@rneui` icon-prop typings clash with React 19 types â€” pass `@expo/vector-icons` elements (e.g. `icon={<FontAwesome .../>}`) instead of `{ name, type }` icon objects.

The project is on Expo SDK 57 (`react-native` 0.86, React 19.2). Google Sign-In requires a real dev-client build (`expo run:ios`/`run:android`) rather than plain Expo Go â€” Expo Go's fixed identity (`host.exp.exponent`) can't satisfy the OAuth redirect URI registered for this app's real bundle ID (`com.korchev.fragrancecollection`).

## Architecture

### Source layout

All source lives under `src/` (`src/app` is auto-detected by Expo Router). Folder and file names are lowercase kebab-case. Imports across folders use the `@/` alias (`@/* â†’ ./src/*`, defined in `tsconfig.json` `paths`; Metro resolves it natively) â€” same-directory imports stay relative (`./delete-bottom-sheet`). Static assets (`assets/`, referenced by `app.json`) and `db/schema.sql` stay at the repo root.

### Routing (`src/app/` â€” Expo Router, file-based)

```
src/app/_layout.tsx                     â€” providers + Stack.Protected auth gate (signed-in vs "sign-in")
src/app/sign-in.tsx                     â€” signed-out screen
src/app/picker.tsx                      â€” slot-machine picker, fullScreenModal opened from the Collection tab's FAB
                                          (must stay fullScreenModal: a pageSheet physically tracks downward drags even with
                                          gestureEnabled: false, stealing the lever's pull; X button closes)
src/app/fragrance-detail.tsx            â€” detail formSheet; param-driven display data, plus editable rating/notes
                                          when opened with a collection row `id` (resolves the live row from
                                          AuthContext's collection query)
src/app/profile.tsx                     â€” account formSheet: stats, wear-reminder toggle, sign out, delete account
src/app/wear-history.tsx                â€” personal wear diary formSheet, grouped by calendar day
src/app/(tabs)/_layout.tsx              â€” NativeTabs root tab bar (Discover / Collection)
src/app/(tabs)/(discover)/_layout.tsx   â€” Discover tab's Stack (header), wraps the nested top-tabs group
src/app/(tabs)/(discover)/(top-tabs)/_layout.tsx + index.tsx + search.tsx â€” Most Worn / Search sub-tabs (js-top-tabs)
src/app/(tabs)/(collection)/_layout.tsx + index.tsx â€” Collection tab (list + picker FAB)
```

Root layout (`src/app/_layout.tsx`) wraps the tree in `QueryClientProvider > ToastContextProvider > AuthProvider > ThemeContextProvider`, then renders `<Stack.Protected guard={!!user}>`/`guard={!user}` to switch between `(tabs)` and `sign-in` â€” this replaces what used to be a manual `user ? <Tabs/> : <SigninScreen/>` check. It also holds an `authLoading` gate (see `src/contexts/auth-context.tsx`) that renders nothing until the initial `supabase.auth.getSession()` resolves, to avoid a sign-in-screen flash on cold start.

**NativeTabs render no header** â€” each tab (`(discover)`, `(collection)`) has its own nested `<Stack>` whose only job is supplying the shared custom `src/components/header.tsx` via `screenOptions.header`, using `getHeaderTitle` from `expo-router/react-navigation` (not `@react-navigation/elements` directly â€” Expo Router blocks direct `@react-navigation/*` imports as of SDK 56+; use the `expo-router/react-navigation` and `expo-router/js-top-tabs` re-exports instead).

The nested "Most Worn"/"Search" sub-tabs under Discover use `expo-router/js-top-tabs`'s `createMaterialTopTabNavigator` wrapped in `withLayoutContext` â€” this is the pattern to follow for any other nested-tab-inside-a-tab section. Route file names (`index.tsx`/`search.tsx`) aren't presentable, so labels come from each `Tabs.Screen`/`MaterialTopTabs.Screen`'s `options.title`, not the route name.

**NativeTabs tradeoffs accepted in this app**: a single `tintColor` (no separate configurable inactive-tab color), no custom `tabBarStyle` (native OS chrome instead â€” iOS liquid glass / Android Material 3), and the Collection tab's custom hand-drawn icon (originally a live SVG component, since deleted) is now a pre-rasterized `assets/collection-icon.png` used via `NativeTabs.Trigger.Icon src={...} renderingMode="template"` (NativeTabs icons can't render arbitrary React components â€” only SF Symbols/Material Symbols/static images/vector-icon-library glyphs).

### Provider stack

- **`src/contexts/auth-context.tsx`** (`useAuth`) â€” owns Supabase auth state (`user`, `authLoading`; the Google `id_token`/Apple `identityToken` are each exchanged via `supabase.auth.signInWithIdToken`), and is the sole owner of all *write* operations for a user's collection: `addFragranceToCollection`, `requestDelete`/`cancelDelete` (deferred delete with toast-undo window), `updateFragrance` (also accepts `rating`/`notes`), `incrementWear` (calls the `increment_wear` RPC â€” atomic counter + `last_worn` stamp + a `wear_events` log row for the Most Worn leaderboard, invalidating the `["top-worn"]`/`["wear-history"]` queries; **one wear per calendar day** â€” the RPC takes the device timezone from `src/lib/utils/worn-today.ts` and returns `false` when today's wear already happened, which surfaces as a toast; wear buttons dim via `isWornToday`; a successful wear shows a toast with an **Undo** action wired to `undoWear`, which calls the `undo_wear` RPC â€” a mistap would otherwise cost the whole calendar day), `deleteAccount` (invokes the `delete-account` edge function, then signs out locally â€” the server-side session is already gone with the deleted user). The collection *read* is a TanStack query (`["collection", user.id]`); writes and Supabase Realtime `postgres_changes` events both invalidate it â€” do not rely on Realtime alone for UI freshness. Exposes `userCollection` (raw), `sortedCollection` (sorted by `times_worn`), `visibleSortedCollection` (minus rows pending deletion â€” screens should render this one), and `collectionPending`/`collectionError` for loading/error UI. The Supabase user object is shimmed with a `photoURL` field (from `user_metadata.avatar_url`) for `src/components/header.tsx` compatibility. Also owns the picker state (the picker UI lives at `src/app/picker.tsx`, a modal opened from the Collection tab's FAB): `frag`/`setFrag`/`index`/`getNewFrag` (weighted via `src/lib/utils/pick-weighted-index.ts` â€” inverse `times_worn`, ~85% downweight for anything worn in the last 24h; `last_worn` is an ISO string, `new Date(...)` it before math). `wear_events.user_fragrance_id` links each event back to the collection row that produced it (`ON DELETE SET NULL`, so leaderboard history survives row deletion) â€” `undo_wear` uses it to find "today's" event precisely.
- **`src/contexts/toast-context.tsx`** (`useToast`) â€” bottom snackbar (`showToast({ message, actionLabel, onAction })`), used for the delete-undo flow. Sits above `AuthProvider` so any context/screen can use it.

### Data fetching (TanStack Query â€” there is no DataContext)

All reads go through TanStack Query (`@tanstack/react-query`; client in `src/lib/query-client.ts`, 5-min staleTime; `onlineManager` is wired to `@react-native-community/netinfo` so queries pause offline and refetch on reconnect, and `focusManager` is wired to `AppState` so returning to the foreground refetches stale queries). Shared hooks live in `src/lib/queries.ts`: `useTopWorn(period)` (community Most Worn leaderboard via the `top_worn_fragrances` RPC; `period` is a `WEAR_PERIODS` key â€” day/week/month/year/all; every period counts `wear_events` (`'all'` is just unbounded), so wears survive collection-row deletion), `useWearHistory(userId)` (personal wear diary, newest first, powers `wear-history.tsx` and the profile stats), `useFragranceSearch(term, filters)` (the `search_fragrances` RPC; enabled once `term` â‰Ą `MIN_SEARCH_LENGTH` **or** the brand filter is set â€” browse mode; the search screen debounces input with `useDebouncedValue` from `src/lib/utils/use-debounced-value.ts` and uses `placeholderData: keepPreviousData` to avoid flicker), `useBrands()` (brand facet list for the filter picker via the `list_brands` RPC, 24h staleTime, filtered client-side), and `useRemindersEnabled(userId)` (whether the profile's wear-reminder toggle should read on â€” `true` unless every one of the user's `user_push_tokens` rows has `reminders_enabled = false`). The detail sheet at `src/app/fragrance-detail.tsx` (presented as a `formSheet` from the root Stack) is display-driven from route params, but when opened with a collection row's `id` param it also resolves the live row from `AuthContext.userCollection` â€” that's what unlocks the editable personal rating/notes and keeps wear stats fresh after a wear. The collection query lives inside `AuthContext` (needs `user.id` + write invalidation). `src/components/empty-state.tsx` is the shared empty/error placeholder (icon + message + optional action) â€” use it for new empty lists or failed queries instead of ad hoc `<Text>`. When adding a new data source, add a hook to `src/lib/queries.ts` rather than fetching in components or creating a new context; queryFns must `throw` on failure so react-query surfaces `error`.
- **`src/contexts/theme-context.tsx`** (`useTheme`) â€” light/dark theme. Precomputes plain-string class groups (`headerColors`, `viewColors`, `cardColors`, `modalColors`) plus `baseColors`/`baseTextClass`/`baseBorderClass` that components destructure and interpolate into `className` strings. **Important**: NativeWind's class scanner only picks up complete literal class-name strings appearing in source â€” a runtime-built string like `` `text-${baseColors}` `` (where `baseColors` is a bare variable) will NOT work, even though it's valid JS. Always use the precomputed `baseTextClass`/`baseBorderClass` fields (or add a new literal-branch field here) instead of interpolating a bare color name into a class string.

`AuthContext` uses `useQueryClient()`/`useQuery()` internally, so `QueryClientProvider` must stay above `AuthProvider` in the tree.

### Styling

NativeWind v4 (`className` props, Tailwind v3 config in `tailwind.config.js`) â€” not `tailwind-rn`. Colors referenced outside of `className` (icon `color` props, RN style objects for things Tailwind can't express like `shadowColor`/`shadowOffset`) use `getColor("green-500")` from `src/lib/utils/colors.ts` (a thin wrapper over `tailwindcss/colors`). Third-party UI components from `@rneui/themed` (ListItem, Chip, SearchBar, BottomSheet, Avatar) generally do **not** support `className` on their style-like props (`containerStyle`/`buttonStyle`/`titleStyle`/etc.) â€” pass real RN style objects (via `getColor()` for theme-driven colors) there instead, not `{ className: "..." }` (that silently no-ops).

### Data model conventions

- Postgres layout (see `db/schema.sql`): `user_fragrances` for a user's collection (`id`, `user_id`, nullable `fragrance_id` FK to the catalog â€” populated for adds from search/leaderboard, null for legacy rows and manual adds, `name`, `image_url`, `times_worn`, `last_worn`, personal `rating` (1â€“5, nullable) and `notes` (â‰¤2000 chars, nullable â€” user-entered, edited on the detail sheet), `created_at`), RLS-scoped to `(select auth.uid())`; `wear_events` for the immutable per-wear log (denormalized `name`/`image_url` snapshots so leaderboard history survives collection-row deletion, plus `user_fragrance_id` â€” `ON DELETE SET NULL` â€” linking back to the originating row so `undo_wear` can find "today's" event precisely; own-rows RLS â€” cross-user aggregation only via the `top_worn_fragrances` SECURITY DEFINER RPC, `EXECUTE` revoked from `anon`).
- `name` is stored as a single `"Brand - Title"` string and split on `" - "` at render time (see `src/components/collection-list-item.tsx`, `src/components/top-list-item.tsx`) rather than stored as separate fields â€” keep this convention when adding new fields/components that display fragrance names.
- List item components (`src/components/collection-list-item.tsx` for the user's collection, `src/components/top-list-item.tsx` for search/leaderboard results) are intentionally separate compositions rather than a single parametrized component â€” the two contexts have diverging fields (`place`/`wearCount` vs. `timesWorn`/`lastWorn`) and actions (add vs. wear/delete). Both are built from the shared compound-card primitives in `src/components/card.tsx`: `Card.Root` (plain row, or swipeable when given `swipeActions`), `Card.Rank`, `Card.Thumbnail`, `Card.Content`, `Card.Title`, `Card.Subtitle`, `Card.WornInfo` (personal "Worn Nx Â· last worn"), `Card.WearCount` (community count), `Card.Overlay` (dark bottom scrim pinned over an image â€” the picker reel's info strip), and `Card.ActionButton` (tinted circular button â€” `variant` maps to the theme's `buttons` group; the icon is a render prop receiving the paired icon color; also reused outside cards, e.g. the picker's overlaid wear button via `size='lg'` + `className`/`style` pass-throughs). `Card.Title`/`Card.Subtitle`/`Card.WornInfo` take an `overlay` prop that switches from themed list-row styling to fixed white-on-scrim styling for use inside `Card.Overlay`. Image-URL validation is shared via `getImageSource` in `src/lib/utils/image-source.ts` (used by `Card.Thumbnail` and the picker reel). Keep the split: add new row types as new compositions of `Card.*`, and put new shared card sections in `card.tsx` rather than duplicating layout. Both push `/fragrance-detail` on row press.
