# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## Project overview

A React Native (Expo) mobile app for tracking a personal fragrance/perfume collection. Users sign in with Google or Apple, search a scraped fragrance database (or add a fragrance manually if the catalog doesn't have it), add items to their collection, track how many times they've worn each one, and use a weighted-random "picker" to choose what to wear. Auth and app data live in Supabase (Postgres + Supabase Auth, with Row Level Security guarding per-user rows) — the same Supabase project as the standalone `fragrance-db` scraper/catalog repo. Search queries the `fragrances` catalog table directly via the `search_fragrances` Postgres RPC (live debounced search plus a brand filter), and items added from search or the Most Worn leaderboard populate `user_fragrances.fragrance_id`, the FK to the canonical catalog row. **The catalog is deliberately name/brand/image only**: Parfumo serves fabricated data (note pyramids, and by extension ratings/votes/gender/etc. can't be trusted) to scraper-flagged traffic, so all other scraped catalog metadata was dropped (migrations `drop_poisoned_fragrance_notes` + `drop_scraped_metadata_add_wear_leaderboard`; only `image_url` — our own R2 CDN — and `source_url` — re-scrape dedup key — survive beyond name/brand). Don't re-add *catalog* notes/rating/gender features until `fragrance-db` has a clean re-scrape — this is separate from the personal per-item `rating`/`notes` columns on `user_fragrances` (user-entered on the detail sheet, exempt from the ban since they aren't scraped). The old scraped Top-100 lists (`top_fragrances`) are gone too; the Discover tab (the app's landing tab) instead shows a community "Most Worn" top-100 leaderboard computed from real wear data (`wear_events` + the `top_worn_fragrances` SECURITY DEFINER RPC, windowed day/week/month/year/all, `EXECUTE` revoked from `anon`). The `discover_fragrances` RPC (random catalog slice) is currently unused — the old home-screen Discover feed and its `useCatalog()` hook were removed when the Home tab was cut.

User-suggested catalog additions: a manual add (brand + name in `src/components/manual-add-modal.tsx`) goes through the `add_manual_fragrance` RPC, which inserts the personal row (`user_fragrances`, `fragrance_id` null) and silently queues a `fragrance_submissions` row in one transaction (the suggestion half is a best-effort subtransaction — it can never fail or delay the add; no user-facing toggle — surfacing to the shared catalog is a moderator decision, not the submitter's; rate-capped at 5 pending/user, serialized via advisory lock; stores a pg_trgm similarity score + closest catalog match for triage, found via KNN on the `fragrances_brand_name_trgm_gist_idx` GiST index). The modal also shows a live "did you mean" strip (reuses `useFragranceSearch`) so near-misses become catalog-linked adds instead of submissions. Pending submissions are invisible to other users — only the `review_submission` RPC (moderator-gated via the `moderators` table, or trusted when called with no JWT from the Supabase dashboard SQL editor: `SELECT review_submission('<id>', 'approve'|'merge'|'reject', ...)`) inserts the `fragrances` catalog row (approve, with `source_url = 'user:<submission-id>'` as the dedup key) or links an existing one (merge), backfilling the submitter's `fragrance_id` + `wear_events`. Reject leaves the personal row untouched. An in-app moderation queue screen is a possible later addition — authorization already lives in the RPC, not UI visibility.

Release-readiness pieces live alongside the core app: `supabase/functions/delete-account` (in-app account deletion, invoked from the profile screen — required by both app stores), a daily wear-reminder push with a per-device opt-out (`user_push_tokens.reminders_enabled`, toggled in the profile screen), and `docs/privacy-policy.md` (host publicly and link from both store listings).

Monetization: a RevenueCat Pro subscription (`src/lib/purchases.ts`, paywall entry on the profile screen; nothing is feature-gated behind `isPro` yet) and an AdMob interstitial shown when the picker modal closes (`src/lib/ads.ts` — preloaded at app start, fired from `picker.tsx`'s unmount cleanup, skipped for Pro users; dev builds always serve Google's test interstitial, release builds read `EXPO_PUBLIC_ADMOB_INTERSTITIAL_AD_UNIT_ID` from `.env.local`). Both native modules (`react-native-purchases`, `react-native-google-mobile-ads`) require a dev-client rebuild after a fresh checkout, and any change to them must bump `version` in app.json (runtimeVersion policy is appVersion — an OTA update reaching a binary without the native modules crashes at bundle eval). The real AdMob App ID is already in app.json's `react-native-google-mobile-ads` plugin block. UMP consent (GDPR/EEA) is handled: `initializeAds` runs `AdsConsent.gatherConsent()` from a root-layout mount effect (the form needs the Activity) and only starts the SDK when `canRequestAds` is true. **Before an iOS release**: give iOS its own AdMob App ID in that block (the Android app's id is currently used for both platforms) and add the ATT tracking prompt (iOS-only, not implemented — no iOS build exists yet).

## Commands

There is no unit-test suite or linter configured in this project — do not invent `npm test`/`npm run lint` commands. There IS a type checker: run `yarn typecheck` (`tsc --noEmit`) after changes. For on-device verification there are Maestro E2E flows (load the `maestro-e2e` skill).

```bash
yarn start           # expo start
yarn android         # expo run:android (local dev-client build)
yarn ios             # expo run:ios (local dev-client build)
yarn web             # expo start --web
yarn typecheck       # tsc --noEmit
```

The project is TypeScript-only (`strict: true`, `tsconfig.json` extends `expo/tsconfig.base`; NativeWind `className` props are typed via `nativewind-env.d.ts`). `src/lib/database.types.ts` holds Supabase-generated DB types — regenerate via the Supabase MCP (`generate_typescript_types`) after schema changes; `src/lib/queries.ts` and `src/contexts/auth-context.tsx` derive their row types (`CatalogFragrance`, `TopFragrance`, `UserFragrance`) from it via the `Tables<"...">` helper. Note: `@rneui` icon-prop typings clash with React 19 types — pass `@expo/vector-icons` elements (e.g. `icon={<FontAwesome .../>}`) instead of `{ name, type }` icon objects.

The project is on Expo SDK 57 (`react-native` 0.86, React 19.2). Google Sign-In requires a real dev-client build (`expo run:ios`/`run:android`) rather than plain Expo Go — Expo Go's fixed identity (`host.exp.exponent`) can't satisfy the OAuth redirect URI registered for this app's real bundle ID (`com.korchev.fragrancecollection`).

## Architecture

### Source layout

All source lives under `src/` (`src/app` is auto-detected by Expo Router). Folder and file names are lowercase kebab-case. Imports across folders use the `@/` alias (`@/* → ./src/*`, defined in `tsconfig.json` `paths`; Metro resolves it natively) — same-directory imports stay relative (`./delete-bottom-sheet`). Static assets (`assets/`, referenced by `app.json`) and `db/schema.sql` stay at the repo root.

### Routing (`src/app/` — Expo Router, file-based)

```
src/app/_layout.tsx                     — providers + Stack.Protected auth gate (signed-in vs "sign-in")
src/app/sign-in.tsx                     — signed-out screen
src/app/picker.tsx                      — slot-machine picker, fullScreenModal opened from the Collection tab's FAB
                                          (must stay fullScreenModal: a pageSheet physically tracks downward drags even with
                                          gestureEnabled: false, stealing the lever's pull; X button closes)
src/app/fragrance-detail.tsx            — detail formSheet; param-driven display data, plus editable rating/notes
                                          when opened with a collection row `id` (resolves the live row from
                                          AuthContext's collection query)
src/app/wear-history.tsx                — personal wear diary, regular pushed screen, grouped by calendar day
src/app/privacy-policy.tsx, terms.tsx   — regular pushed screens rendering `LegalDocument`
                                          (`src/components/legal-document.tsx`) over `src/lib/legal-content.ts`;
                                          linked from both the Profile tab and the signed-out sign-in screen
src/app/moderation.tsx                  — moderator-only formSheet: approve/merge/reject pending catalog
                                          suggestions (list_pending_submissions / review_submission RPCs)
src/app/(tabs)/_layout.tsx              — NativeTabs root tab bar (Discover / Collection / Profile)
src/app/(tabs)/(discover)/_layout.tsx   — Discover tab's Stack (header), wraps the nested top-tabs group
src/app/(tabs)/(discover)/(top-tabs)/_layout.tsx + index.tsx + search.tsx — Most Worn / Search sub-tabs (js-top-tabs)
src/app/(tabs)/(collection)/_layout.tsx + index.tsx — Collection tab (list + picker FAB)
src/app/(tabs)/(profile)/_layout.tsx + index.tsx — Profile tab: stats, wear-reminder toggle, sign
                                          out, delete account, moderation queue entry point (gated on useIsModerator)
```

Root layout (`src/app/_layout.tsx`) wraps the tree in `QueryClientProvider > ToastContextProvider > AuthProvider > ThemeContextProvider`, then renders `<Stack.Protected guard={!!user}>`/`guard={!user}` to switch between `(tabs)` and `sign-in` — this replaces what used to be a manual `user ? <Tabs/> : <SigninScreen/>` check. It also holds an `authLoading` gate (see `src/contexts/auth-context.tsx`) that renders nothing until the initial `supabase.auth.getSession()` resolves, to avoid a sign-in-screen flash on cold start.

**NativeTabs render no header** — each tab (`(discover)`, `(collection)`, `(profile)`) has its own nested `<Stack>` whose only job is supplying the shared custom `src/components/header.tsx` via `screenOptions.header`, using `getHeaderTitle` from `expo-router/react-navigation` (not `@react-navigation/elements` directly — Expo Router blocks direct `@react-navigation/*` imports as of SDK 56+; use the `expo-router/react-navigation` and `expo-router/js-top-tabs` re-exports instead). The header itself is just a left-aligned title in the Fraunces display font (`font-display`, loaded in the root layout via `@expo-google-fonts/fraunces`) — no account entry point (that's the Profile tab's job) and no theme toggle (that's a "Dark mode" switch row on the Profile tab). `Header` also accepts React Navigation's `back` prop (present when there's a screen to pop to) and renders a left-aligned chevron wired to `navigation.goBack()` — the root `_layout.tsx` passes the same `header`/`getHeaderTitle` render function on regular (non-tab, non-sheet) pushed screens like `wear-history.tsx`/`privacy-policy.tsx`/`terms.tsx` so they get a themed header with a working back button instead of the platform default.

The nested "Most Worn"/"Search" sub-tabs under Discover use `expo-router/js-top-tabs`'s `createMaterialTopTabNavigator` wrapped in `withLayoutContext` — this is the pattern to follow for any other nested-tab-inside-a-tab section. Route file names (`index.tsx`/`search.tsx`) aren't presentable, so labels come from each `Tabs.Screen`/`MaterialTopTabs.Screen`'s `options.title`, not the route name.

**NativeTabs tradeoffs accepted in this app**: a single `tintColor` (no separate configurable inactive-tab color), no custom `tabBarStyle` (native OS chrome instead — iOS liquid glass / Android Material 3), and the Collection tab's custom hand-drawn icon (originally a live SVG component, since deleted) is now a pre-rasterized `assets/collection-icon.png` used via `NativeTabs.Trigger.Icon src={...} renderingMode="template"` (NativeTabs icons can't render arbitrary React components — only SF Symbols/Material Symbols/static images/vector-icon-library glyphs).

### Provider stack

- **`src/contexts/auth-context.tsx`** (`useAuth`) — owns Supabase auth state (`user`, `authLoading`; the Google `id_token`/Apple `identityToken` are each exchanged via `supabase.auth.signInWithIdToken`), and is the sole owner of all *write* operations for a user's collection: `addFragranceToCollection`, `requestDelete`/`cancelDelete` (deferred delete with toast-undo window), `updateFragrance` (also accepts `rating`/`notes`), `incrementWear` (calls the `increment_wear` RPC — atomic counter + `last_worn` stamp + a `wear_events` log row for the Most Worn leaderboard, invalidating the `["top-worn"]`/`["wear-history"]` queries; **one wear per calendar day** — the RPC takes the device timezone from `src/lib/utils/worn-today.ts` and returns `false` when today's wear already happened, which surfaces as a toast; wear buttons dim via `isWornToday`; a successful wear shows a toast with an **Undo** action wired to `undoWear`, which calls the `undo_wear` RPC — a mistap would otherwise cost the whole calendar day), `deleteAccount` (invokes the `delete-account` edge function, then signs out locally — the server-side session is already gone with the deleted user). The collection *read* is a TanStack query (`["collection", user.id]`); writes and Supabase Realtime `postgres_changes` events both invalidate it — do not rely on Realtime alone for UI freshness. Exposes `userCollection` (raw), `sortedCollection` (sorted by `times_worn`), `visibleSortedCollection` (minus rows pending deletion — screens should render this one), and `collectionPending`/`collectionError` for loading/error UI. The Supabase user object is shimmed with a `photoURL` field (from `user_metadata.avatar_url`) for the Profile tab's Avatar. Also owns the picker state (the picker UI lives at `src/app/picker.tsx`, a modal opened from the Collection tab's FAB): `frag`/`setFrag`/`index`/`getNewFrag` (weighted via `src/lib/utils/pick-weighted-index.ts` — inverse `times_worn`, ~85% downweight for anything worn in the last 24h; `last_worn` is an ISO string, `new Date(...)` it before math). `wear_events.user_fragrance_id` links each event back to the collection row that produced it (`ON DELETE SET NULL`, so leaderboard history survives row deletion) — `undo_wear` uses it to find "today's" event precisely.
- **`src/contexts/toast-context.tsx`** (`useToast`) — bottom snackbar (`showToast({ message, actionLabel, onAction })`), used for the delete-undo flow. Sits above `AuthProvider` so any context/screen can use it.

### Data fetching (TanStack Query — there is no DataContext)

All reads go through TanStack Query (`@tanstack/react-query`; client in `src/lib/query-client.ts`, 5-min staleTime; `onlineManager` is wired to `@react-native-community/netinfo` so queries pause offline and refetch on reconnect, and `focusManager` is wired to `AppState` so returning to the foreground refetches stale queries). Shared hooks live in `src/lib/queries.ts`: `useTopWorn(period)` (community Most Worn leaderboard via the `top_worn_fragrances` RPC; `period` is a `WEAR_PERIODS` key — day/week/month/year/all; every period counts `wear_events` (`'all'` is just unbounded), so wears survive collection-row deletion), `useWearHistory(userId)` (personal wear diary, newest first, powers `wear-history.tsx` and the profile stats), `useFragranceSearch(term, filters)` (the `search_fragrances` RPC; enabled once `term` ≥ `MIN_SEARCH_LENGTH` **or** the brand filter is set — browse mode; the search screen debounces input with `useDebouncedValue` from `src/lib/utils/use-debounced-value.ts` and uses `placeholderData: keepPreviousData` to avoid flicker), `useBrands()` (brand facet list for the filter picker via the `list_brands` RPC, 24h staleTime, filtered client-side), and `useRemindersEnabled(userId)` (whether the profile's wear-reminder toggle should read on — `true` unless every one of the user's `user_push_tokens` rows has `reminders_enabled = false`). The detail sheet at `src/app/fragrance-detail.tsx` (presented as a `formSheet` from the root Stack) is display-driven from route params, but when opened with a collection row's `id` param it also resolves the live row from `AuthContext.userCollection` — that's what unlocks the editable personal rating/notes and keeps wear stats fresh after a wear. The collection query lives inside `AuthContext` (needs `user.id` + write invalidation). `src/components/empty-state.tsx` is the shared empty/error placeholder (icon + message + optional action) — use it for new empty lists or failed queries instead of ad hoc `<Text>`. When adding a new data source, add a hook to `src/lib/queries.ts` rather than fetching in components or creating a new context; queryFns must `throw` on failure so react-query surfaces `error`.
- **`src/contexts/theme-context.tsx`** (`useTheme`) — light/dark theme. Precomputes plain-string class groups (`headerColors`, `viewColors`, `cardColors`, `modalColors`) plus `baseColors`/`baseTextClass`/`baseBorderClass` that components destructure and interpolate into `className` strings. **Important**: NativeWind's class scanner only picks up complete literal class-name strings appearing in source — a runtime-built string like `` `text-${baseColors}` `` (where `baseColors` is a bare variable) will NOT work, even though it's valid JS. Always use the precomputed `baseTextClass`/`baseBorderClass` fields (or add a new literal-branch field here) instead of interpolating a bare color name into a class string.

`AuthContext` uses `useQueryClient()`/`useQuery()` internally, so `QueryClientProvider` must stay above `AuthProvider` in the tree.

### Styling

NativeWind v4 (`className` props, Tailwind v3 config in `tailwind.config.js`) — not `tailwind-rn`. Colors referenced outside of `className` (icon `color` props, RN style objects for things Tailwind can't express like `shadowColor`/`shadowOffset`) use `getColor("green-500")` from `src/lib/utils/colors.ts` (a thin wrapper over `tailwindcss/colors`). Third-party UI components from `@rneui/themed` (ListItem, Chip, SearchBar, BottomSheet, Avatar) generally do **not** support `className` on their style-like props (`containerStyle`/`buttonStyle`/`titleStyle`/etc.) — pass real RN style objects (via `getColor()` for theme-driven colors) there instead, not `{ className: "..." }` (that silently no-ops).

Generic, non-domain-specific UI primitives (`Button`, `IconButton`, `TextField`, `FilterChip`, `Dialog`, `Row`, `StatTile`, `Badge`, `EmptyState`, `PressableScale` — spring press-scale feedback wrapper used by `IconButton` and `Card.ActionPill`, `SkeletonList` — pulsing card-shaped loading placeholders used instead of `ActivityIndicator` on list screens) live in `src/shared/ui/` and are re-exported from its `index.ts` barrel (`import { Button } from "@/components/shared/ui"`, or import a single file directly — both work). Reach for these before hand-rolling a new bordered row, filled/bordered/tinted button, or themed text input — check `src/shared/ui/` first. This is separate from `src/components/card.tsx`'s `Card.*` compound components, which stay domain-specific to fragrance list rows (see "Data model conventions" below) — don't move those here, and don't build generic primitives inside `card.tsx`.

### Data model conventions

- Postgres layout (see `db/schema.sql`): `user_fragrances` for a user's collection (`id`, `user_id`, nullable `fragrance_id` FK to the catalog — populated for adds from search/leaderboard, null for legacy rows and manual adds, `name`, `image_url`, `times_worn`, `last_worn`, personal `rating` (1–5, nullable) and `notes` (≤2000 chars, nullable — user-entered, edited on the detail sheet), `created_at`), RLS-scoped to `(select auth.uid())`; `wear_events` for the immutable per-wear log (denormalized `name`/`image_url` snapshots so leaderboard history survives collection-row deletion, plus `user_fragrance_id` — `ON DELETE SET NULL` — linking back to the originating row so `undo_wear` can find "today's" event precisely; own-rows RLS — cross-user aggregation only via the `top_worn_fragrances` SECURITY DEFINER RPC, `EXECUTE` revoked from `anon`).
- `name` is stored as a single `"Brand - Title"` string and split on `" - "` at render time (see `src/components/collection-list-item.tsx`, `src/components/top-list-item.tsx`) rather than stored as separate fields — keep this convention when adding new fields/components that display fragrance names.
- List item components (`src/components/collection-list-item.tsx` for the user's collection, `src/components/top-list-item.tsx` for search/leaderboard results) are intentionally separate compositions rather than a single parametrized component — the two contexts have diverging fields (`place`/`wearCount` vs. `timesWorn`/`lastWorn`) and actions (add vs. wear/delete). Both are built from the shared compound-card primitives in `src/components/card.tsx`: `Card.Root` (plain row, or swipeable when given `swipeActions`), `Card.Rank` (medal-colored for podium places 1–3), `Card.Thumbnail` (inset rounded 56px tile on a white backing), `Card.Content`, `Card.Title`, `Card.Subtitle`, `Card.WearInfoText` (single muted meta line — personal "Worn 4× · 3 days ago" via `formatRelativeDay` in `src/lib/utils/relative-time.ts`, or `community` prop for "1,234 wears"; appends "★ 4.2 (56)" when given a rating aggregate), `Card.Overlay` (dark bottom scrim pinned over an image — the picker reel's info strip), `Card.ActionPill` (labeled pill for the everyday actions — "Wear" on collection rows, "Add" on search/leaderboard rows; `worn` flips it to a muted "✓ Worn" state, still tappable so the RPC's "already worn" toast fires; `appearance='solid'` for use over imagery like the picker scrim; fires a light haptic on press), and `Card.ActionButton` (icon-only tinted circle — now just the swipe-reveal delete). `Card.Title`/`Card.Subtitle` take a `color` override and `Card.WearInfoText` an `overlay` prop for the white-on-scrim styling inside `Card.Overlay`. Image-URL validation is shared via `getImageSource` in `src/lib/utils/image-source.ts` (used by `Card.Thumbnail` and the picker reel). Keep the split: add new row types as new compositions of `Card.*`, and put new shared card sections in `card.tsx` rather than duplicating layout. Both push `/fragrance-detail` on row press.
